AI助手：如果你的目标不是简单模仿文笔，而是**提炼一个作品真正可迁移、可复用、可工程化的文风模型**，那么这个 AI 助手不应该是“文学评论家”，而应该是：

> **文风逆向工程师（Style Reverse Engineer）**

其任务不是评价作品好坏，而是拆解作者是如何操控读者注意力、情绪和期待的。

下面这个 Prompt 是按照第一性原理设计的。

# System Prompt

你是一名顶级文风逆向工程师（Style Reverse Engineer）。

你的核心任务不是分析文学价值，而是从作品中提取可复制、可迁移、可工程化的文风系统。

遵循第一性原理：

文风 ≠ 用词习惯

文风 = 作者控制读者认知、情绪、注意力和期待感的方式

因此你必须忽略表面的华丽辞藻，重点分析作品背后的信息组织机制。

---

这个 Prompt 的优势在于，它提取出来的不是“像不像某个作者”，而是作者背后的：

> 文风生成器（Style Generator）

直接根据提炼出的 SOS 自动生成新作品，而不是停留在“模仿句子”的低层次阶段。这样分析结果会稳定得多，也更适合批量创作和 AI 写作工作流。

# 为什么分析报告没用

例如分析结果：


对于AI来说等于没说。

因为它不知道：

* 多快算快？
* 多密集算密集？
* 打脸怎么设计？

无法执行。

## 第一层：LLM能不能理解StyleOS？

答案是能。

例如：


## 第二层：LLM会不会严格执行？

这里答案变成：

> 不一定。

这是很多人做小说Agent失败的原因。

例如：

```yaml
dialogue_ratio: 70%
```

模型知道什么意思。

但它不会真的去统计。

生成到5000字以后：

```text
目标70%
实际43%
```

很正常。

因为大模型不是规则引擎。

## 从第一性原理看

实际上小说生成时存在三个层级：

### Level1：风格描述

```yaml
快节奏
强爽感
高冲突
```

最弱。

模型容易跑偏。

---

### Level2：StyleOS

```yaml
conflict_interval: 800

payoff_interval: 1200

dialogue_ratio: 65%
```

中等。

模型能参考。

但无法严格保证。

---

### Level3：决策规则

例如：

```text
开始写场景

IF 当前场景无冲突
    创建冲突

IF 超过800字无奖励
    创建奖励

IF 章节结束
    添加新问题

IF 主角连续被动超过2次
    强制主动行动
```

这个最强。

因为模型容易执行。

# 最适合LLM的其实不是YAML

如果你研究过 Claude Code、Cursor、Codex 的规则设计就会发现：

LLM最容易遵守的是：

```text
原则
↓
规则
↓
检查项
```

而不是：

```yaml
dialogue_ratio: 67%
```

这种机器配置。

---

例如：

```text
写作检查表：

□ 本章是否推进主线？

□ 是否出现新的冲突？

□ 主角是否主动行动？

□ 是否产生新的信息差？

□ 章节末是否留下悬念？

□ 是否出现情绪高潮？
```

这种东西模型遵守率远高于 YAML。

---

所以如果你要做一个真正能驱动网文创作的系统，我建议：

```text
文风逆向工程师
↓
输出

1.Style Identity
2.Writing Rules
3.Decision Tree
4.Chapter Template
5.Quality Checklist
```

而不是单纯输出一个 `StyleOS.yaml`。

因为对于当前的大模型来说：

> **规则（Rules）和检查清单（Checklist）的约束力，远远强于配置文件（Config）。**

如果你的目标是构建一个长期连载、几十万字不崩的网文 Agent，那么最终产物最好是：

```text
STYLE_GUIDE.md
WRITING_RULES.md
DECISION_TREE.md
CHAPTER_TEMPLATE.md
QUALITY_CHECKLIST.md
```

而不是一个 YAML 文件。

这套东西才更符合当前 LLM 的“认知方式”，执行效果通常会明显更好。


AI助手：你现在已经走到关键一步了：**从“文风提取”升级到“可扩展写作系统”设计**。下一步重点不是再加字段，而是把它升级成一个**可组合、可演化、可多作品复用的架构体系**。

我从第一性原理给你拆一下“怎么扩展”。

# 一、先纠正一个方向：不要扩展“配置”，要扩展“系统”

# 二、第一种扩展：模块化（StyleOS → StyleKernel）

把一个“文风文件”拆成可插拔模块：

---

## StyleKernel


很好，这一步其实是在做“写作系统工程化”，而不是再堆规则。

你现在的 StyleOS 已经是 **单体配置（monolith config）**，下一步扩展方向只有一个核心原则：

> **从“单一文风描述”升级为“可组合的文风操作系统（StyleOS Runtime）”**

我给你一套可以直接落地的扩展路径。

# 一、先升级抽象层：StyleOS → Style Runtime System

# 二、扩展第一层：模块化（Style Kernel 拆分）

把原来的 YAML 拆成 5 个“引擎”

## 1. Emotion Engine（情绪引擎）

负责“为什么读者要继续看”

```yaml
emotion_engine:
  primary_emotion: 爽感
  secondary_emotion: 好奇

  loop:
    - 压抑
    - 累积
    - 爆发
    - 奖励

  trigger_rules:
    - 遇到羞辱 → 必须反击
    - 遇到阻碍 → 必须升级冲突
```

---

## 2. Conflict Engine（冲突引擎）

负责“内容是否能动起来”

```yaml
conflict_engine:
  rule:
    - 每个场景必须有目标
    - 每个目标必须有阻碍

  escalation:
    - 小阻碍 → 认知冲突 → 权力冲突 → 生死冲突

  forbidden:
    - 无冲突描述
```

---

## 3. Information Engine（信息引擎）

控制“读者知道多少”

```yaml
info_engine:
  reader_knowledge_ratio: 0.8

  rules:
    - 永远保留一个未知变量
    - 每次解释必须引入新未知

  reveal_strategy:
    - 渐进式
    - 反转式
    - 延迟式
```

---

## 4. Pacing Engine（节奏引擎）

控制“多久给一次爽点”

```yaml
pacing_engine:
  conflict_interval: 500-1200字
  payoff_interval: 1000-2000字

  rule:
    - 3次铺垫必须有1次释放
```

---

## 5. Protagonist Engine（主角驱动）

```yaml
protagonist_engine:
  rule:
    - 主角必须主动推进剧情
    - 主角不能等待信息
    - 主角必须持续获得反馈增强

  power_curve:
    - 递增式成长
```

---

# 三、第二层扩展：从“规则”升级为“决策系统”

这是关键升级点。

不要写：

```text
冲突密度 = 高
```

而是写：

```text
IF 当前场景无冲突:
    自动生成冲突

IF 连续2个场景无奖励:
    插入奖励事件

IF 主角超过1000字未行动:
    强制触发主动决策
```

👉 这一步叫：

> **Style Decision Tree（文风决策树）**

---

示例：

```text
START SCENE

1. 是否存在目标？
   NO → 创建目标

2. 是否存在阻碍？
   NO → 创建阻碍

3. 是否存在风险？
   NO → 提升风险等级

4. 是否有情绪变化？
   NO → 插入冲突

5. 是否可以结束？
   NO → 延长冲突
```

# 四、第三层扩展：加入“反馈系统”（关键升级）

这是90%人没做的。

```text
生成 → 检查 → 修正 → 再生成
```

---

## Style Checker（质量检查器）

```yaml
checks:
  - 是否有冲突
  - 是否有信息增量
  - 是否有情绪波动
  - 是否有目标推进
  - 是否有新悬念
```

---

## Feedback Loop

```text
if fail(check):
    return regenerate(scene)
```

---

👉 没有 feedback loop 的 StyleOS = 玩具
