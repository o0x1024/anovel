# 小说创作辅助系统 完整需求与开发方案

> **文档概述**：本文档为 AI 主导创作桌面应用（Electron桌面版）的完整开发方案。系统定位为 **AI 主导创作、作者提供方向**——作者负责灵感方向与关键决策，AI 负责从扩写、设定、大纲到正文的全流程内容生成。覆盖产品定位、核心工作流、功能模块、技术架构、数据库设计及落地路径。

---

## 第一部分：产品概述

### 1.1 产品定位

面向小说创作者的 **AI 主导创作系统**，核心模式为 **"作者提供方向，AI 负责内容"**。作者的角色从"执笔者"转变为"方向制定者 + 决策审核者"——提供灵感、大岗、关键场景锚点与品味判断；AI 负责将方向转化为完整作品，执行扩写、设定构建、结构规划、情节填充与正文输出。系统确保作者的创作主权不因 AI 的多层传导而衰减，每一次生成都精确对齐作者的意图。

### 1.2 目标用户

| 用户类型 | 特征与需求 |
|---------|-----------|
| **新手小说创作者** | 缺乏大纲搭建、情节设计经验，需要AI辅助完成基础框架搭建 |
| **职业/资深作者** | 有明确创作思路，需要AI提升内容产出效率，辅助填充细节、优化情节 |
| **兼职碎片化创作者** | 时间碎片化，需要支持断点创作、内容回溯，随时调整创作进度 |

### 1.3 核心价值

1. **创作主权不衰减**：作者只需提供灵感方向与大岗，AI 负责全流程内容产出；系统通过锚点注入、意图对齐、异常驱动审批等机制，确保作者的核心意图在 AI 多层传导后不扭曲、不稀释
2. **极致产出效率**：作者从"逐字写作"中解放，专注方向决策与品味判断，AI 承担所有执行层工作，创作效率提升一个数量级
3. **品味沉淀与复用**：系统持续学习作者的决策偏好与审美倾向，越用越懂作者，跨作品复用创作基因，让第二本书的起点远高于第一本
4. **超越均值化**：内置反均值化引擎，主动规避 AI 生成内容的"最安全答案"倾向，确保每部作品保持独特性和意外感

## 第二部分：核心创作工作流

> **核心模式**：作者提供方向，AI 负责全流程内容执行。作者角色为"方向制定者 + 决策审核者"，AI 角色为"执行创作者"。所有步骤支持暂停/继续、回退、重新生成、微指令干预。

```
创作前置       步骤1 → 步骤2 → 步骤3 → 步骤4 → 步骤5 → 步骤6
大岗孵化器 → 初始想法 → 核心设定 → 分卷大纲 → 分章情节 → 章节大纲 → 正文生成
   ↓            ↓          ↓         ↓         ↓         ↓         ↓
锚点注入 ←→ [贯穿全流程] ←→ 过程干预通道 ←→ [分层决策 & 异常驱动审批]
```

### 创作前置阶段：大岗孵化器

> 在进入正式创作流程之前，帮助作者穷尽思路、验证方向、锁定核心锚点。这一步决定整本书的基因质量。

| 功能 | 说明 |
|------|------|
| **大岗诊断** | 作者输入初步想法后，AI 不急于扩写，先做方向诊断：核心吸引力是什么？市场上类似故事有哪些？差异化在哪里？潜在风险是什么？ |
| **大岗变体探索** | 系统性地探索可能性空间，按维度生成变体矩阵——"如果主角性别互换""如果时代背景改为现代""如果核心冲突从外部压迫改为内在挣扎""如果叙事视角从第三人称改为第一人称"——帮助作者在提交方向前穷尽可能性 |
| **大岗对标分析** | 作者可指定对标作品组合（如"《甄嬛传》的宫斗密度 + 《琅琊榜》的智谋深度"），AI 分析对标作品的核心要素，基于作者大岗自动融合生成方向建议 |
| **关键场景锚点收集** | 在这一阶段收集作者脑海中不可妥协的核心画面——"我必须有一个场景：主角在雨中跪在仇人门前""结局必须是对手成为战友"——这些锚点将在后续所有生成环节中被强制保留 |
| **情感基调定调** | 作者标注故事的核心情绪基调（如"温柔的悲凉""压抑中爆发的爽快""疏离的冷峻"），成为全流程生成的情感约束条件 |
| **倒推大纲法** | 作者可先锚定高潮/结局（如"最终BOSS是最初的导师"），AI 反向推导故事骨架：从结局倒推关键转折→从转折倒推铺垫→从铺垫倒推开局，生成\"倒推大纲树\"。此方法天然迫使伏笔提前布局，避免\"先射箭再画靶\" | 
| **锁链式嵌套预览** | 基于倒推结果，生成 A1-A2-B1-A3-B2-C1-B3 式嵌套结构预览，展示事件如何环环相扣、如何在上一事件未结束时提前埋入下一事件的伏笔，确保期待感不断链 |
| **输出产物** | 确认后的扩写方向 + 锚点清单 + 情感基调声明 + 倒推大纲树 + 嵌套结构预览，作为后续所有步骤的创作宪法 |

### 锚点注入机制（贯穿全流程）

> 锚点是作者核心意图的"钉子"，钉在创作版图上不可移动。AI 在钉子之间自由发挥，但不能拔掉钉子。
>
> **番茄方法论启发**：锚点的本质是让角色/情节在读者心中留下"记忆标签"。一个好的锚点能让读者在万千小说中精准回忆起你的角色和剧情。

| 锚点类型 | 说明 | 示例 |
|----------|------|------|
| **场景锚点** | 必须出现的关键场景/画面 | "主角必须在暴雨中跪在仇人门前" |
| **角色锚点** | 角色不可更改的核心特质或命运 | "反派从头到尾不能洗白""某个角色必须在第三卷牺牲" |
| **情节锚点** | 不可更改的情节走向 | "第一卷结尾必须揭示主角身份""最终 BOSS 必须是最初的导师" |
| **情感锚点** | 必须达成的情感效果 | "读者必须在揭露真相那一刻感到强烈的心疼" |
| **结构锚点** | 叙事结构的硬约束 | "全书必须是双线叙事，双线在最后一章交汇" |
| 🆕 **记忆锚点** | 人物独特的记忆标签（外貌/行为/语言/时间），确保角色辨识度 | "右手臂20厘米刀疤""每逢月圆夜功力尽失""口头禅：轮到我十步杀一人了" |
| 🆕 **反差锚点** | 人物内外的矛盾张力，增加角色立体感 | "娇小女生武器是比她还高的铁锤""战力天花板却是生活白痴""硬汉人设口头禅是'讨厌死了啦'" |

**锚点工作机制**：

1. **锚点注入入口**：在大岗孵化器阶段集中收集锚点，同时在每个创作步骤中允许作者追加新锚点（如写到某一章时触发灵感，随时新增锚点）
2. **锚点自动扫描**：每一步生成前，AI 自动读取全部锚点清单，在 Prompt 中注入锚点约束
3. **锚点完整性报告**：每一步生成后，系统自动检测锚点是否被遗漏或扭曲，生成对比报告——"你的'雨中跪仇人门'锚点已体现在第3章第2节点，但情感强度偏离了你设定的基调"
4. **锚点冲突检测**：当新锚点与已有设定/大纲冲突时，系统主动提示并要求作者决策——是调整设定以容纳锚点，还是放弃锚点

### 过程干预通道（贯穿全流程）

> 作者可在 AI 生成过程中随时插入微指令，在 AI 跑偏的第一时间纠正，而非等整章生成完毕后才返工。

| 干预方式 | 说明 |
|----------|------|
| **实时方向调整** | AI 正在生成某节点内容时，作者可随时暂停并输入指令："停，这个角色不应该在这里出现，换一个方向"——AI 丢弃当前节点的生成结果，基于调整后的方向立即重新生成 |
| **微指令注入** | 在任何生成结果上，作者可输入一句话调整指令："把这段对话改得更紧张""多用短句，加快节奏""给场景增加雨天的氛围"——AI 局部重写而非全部重来 |
| **条件式预设** | 作者预设全局规则，AI 生成时自动遵守："任何角色死亡的情节必须暂停并询问我""任何吻戏跳过""任何章节中的情节节点不超过5个" |
| **A/B 流生成** | 在关键分叉点，作者不确定走哪个方向时，AI 同时生成 A/B 两个版本的短内容（约500字），作者选择后 AI 基于选中版本继续 |
| **否决即学习** | 作者否决某次生成结果时，系统记录否决原因（可选标签：太俗套/不符合角色/节奏太慢/对话不自然/偏离锚点），AI 立即按原因重新生成，同时该偏好进入品味档案 |

### 步骤1：初始想法扩写

| 维度 | 说明 |
|------|------|
| **输入** | 大岗孵化器的输出：确认后的创作方向 + 锚点清单 + 情感基调声明 |
| **输出** | AI 生成3-5个不同方向的扩写版本，每个版本包含：核心亮点、核心冲突、故事基调、受众定位、**与锚点的对齐度评分** |
| **操作权限** | 选择某个版本、手动修改扩写内容、微指令调整方向、追加锚点 |
| **AI 自检** | 扩写完成后，AI 切换为编辑模式自评：每个版本的独特性评分（是否落入俗套）、与对标作品的重合度 |
| **下一步** | 用户确认扩写内容后，进入核心设定生成环节 |

### 步骤2：核心设定生成

| 维度 | 说明 |
|------|------|
| **输入** | 确认后的扩写内容 + 锚点清单 + 用户手动补充的设定要求 |
| **输出** | 结构化核心设定包：**人物设定**（姓名/年龄/身份/性格特征/口头禅/内在矛盾/行为逻辑/关系图谱/角色弧线）、**世界观设定**（背景/时代/地域/规则体系/层级嵌套）、**核心冲突设定**（主线/副线/核心矛盾/冲突升级路径） |
| **操作权限** | 单独修改任意设定项、新增/删除设定、追加锚点、微指令干预 |
| **AI 自检** | 编辑模式评估：角色立体度（是否有内在矛盾与行为逻辑）、世界观规则是否自洽且可验证、锚点是否全部被设定覆盖 |
| **下一步** | 用户确认核心设定后，进入分卷大纲生成环节 |

### 步骤3：分卷大纲生成

| 维度 | 说明 |
|------|------|
| **输入** | 已确认的核心设定 + 锚点清单 + 用户指定的分卷数量（默认3-5卷） |
| **输出** | 每卷包含：卷名、核心主题、核心冲突、起始终止时间点、**叙事结构节点**（如三幕剧中该卷对应哪一幕）、主要事件节点、卷结尾钩子/反转点、**锚点在该卷的体现位置** |
| **操作权限** | 调整分卷数量、修改任意卷内容、调整分卷顺序、新增/删除分卷、微指令干预 |
| **AI 自检** | 编辑模式评估：卷间节奏分布是否合理、钩子是否有效、锚点分布是否均匀、冲突递进是否有因果链断裂 |
| **下一步** | 用户确认分卷大纲后，进入单卷分章情节生成环节 |

### 步骤4：单卷分五章情节生成

| 维度 | 说明 |
|------|------|
| **输入** | 选中的某一卷分卷大纲 + 锚点清单 + 用户补充的情节要求 |
| **输出** | 该卷下5章的情节大纲，每章包含：核心情节、出场人物、关键冲突点、转折点、结尾钩子、**因果链标注**（本章事件的起因是上一章的哪个事件，本章事件将引发下一章的哪个事件）、与主线/副线关联点 |
| **操作权限** | 调整每章情节内容、调整章节顺序、新增/删除章节、微指令干预 |
| **AI 自检** | 编辑模式评估：因果链是否完整清晰、巧合事件比例（超过20%预警）、**惊喜度评分**（每章情节走向的意外程度1-10，低于阈值自动生成替代版本）、伏笔埋设是否到位 |
| **下一步** | 用户确认5章情节大纲后，进入单章节详细大纲生成环节 |

### 步骤5：单章节详细大纲生成

| 维度 | 说明 |
|------|------|
| **输入** | 选中的某一章情节大纲 + 锚点清单 + 用户补充的章节细节要求 |
| **输出** | 单章节分节点详细大纲（3-5个情节节点），每个节点标注：出场人物、对话要点、场景描写要点、**读者信息差状态**（此时读者知道什么 vs 角色知道什么）、情感强度等级、与前文伏笔的关联 |
| **操作权限** | 调整节点数量、修改任意节点内容、新增/删除节点、微指令干预、追加锚点 |
| **AI 自检** | 编辑模式评估：节点间节奏把控、情感强度曲线是否合理（不应为直线）、伏笔是否有对应回收节点 |
| **下一步** | 用户确认章节详细大纲后，进入正文生成环节 |

### 步骤6：正文内容生成

| 维度 | 说明 |
|------|------|
| **输入** | 单章节详细大纲 + 文风指纹约束 + 叙事记忆体（角色当前状态、前文伏笔、世界观约束） + 用户指定的字数要求（默认每章2000-3000字） |
| **输出** | 完整的章节正文内容，严格匹配文风指纹、符合核心设定、贴合大纲情节 |
| **操作权限** | 微指令局部重写（"这段对话更紧张""用更多短句""增加雨天氛围"）、段落级调整、A/B流选择、手动编辑任意内容 |
| **AI 自检** | 编辑模式评估：角色声音一致性（每个角色的对话是否符合其档案中的口头禅和句式特征）、AI痕迹检测、文风指纹偏差度、与前文的连贯性 |
| **流程结束** | 用户确认正文后该章节创作完成；系统自动更新叙事记忆体（角色状态、伏笔回收状态、时间线），为该作品下一章节生成做好准备 |

### 分层决策与异常驱动审批

> 避免作者陷入"每步都要审"的橡皮图章疲劳。将决策按重要程度分级，同质决策批量处理，正常内容静默通过。

| 决策层级 | 触发条件 | 作者介入方式 |
|----------|----------|-------------|
| **静默通过** | AI 自检全部指标正常（角色一致性≥8分、因果链完整、锚点对齐、惊喜度≥阈值） | 无需作者操作，系统标记为"自动通过"，作者可随时抽查 |
| **批量审阅** | 同质决策合并展示（如5章情节大纲一次性列出），AI 标注每项的置信度 | 作者扫视全局，只修改不满意的项，其余默认通过 |
| **异常提审** | AI 自检触发预警（惊喜度<4分、巧合比例>30%、因果断裂、角色行为偏离档案、锚点未对齐） | 系统高亮问题点并附AI批判意见，作者做出决策 |
| **关键帧决策** | 每卷核心转折、结局、重要角色命运等关键节点 | 必须作者明确确认才能继续，不允许静默通过 |

### 创作节奏模型：压→缓→压→爆发 🆕

> **番茄方法论启发**：情绪曲线应像海浪——持续的压抑让读者窒息，连续的高潮让读者疲劳。经典节奏是"先压，再缓，再压，然后爆发"，每一轮爆发后留下新的钩子，形成期待感的无限循环。

| 节奏阶段 | 说明 | 系统支持 |
|----------|------|---------|
| **压（情绪积累）** | 通过生存环境、不公平待遇、逐步逼近的威胁来制造压抑感 | AI 在分章大纲阶段自动标注每章的情绪强度（1-10），并在全卷视角下检测情绪曲线是否过于平直 |
| **缓（情绪释放）** | 温馨日常、结算时刻、角色间的小互动，让读者有喘息空间 | AI 在大高潮后自动建议插入"结算时刻"（如经历生死后的日常小变化、配角打脸反应） |
| **二次压（铺垫升级）** | 看似好转实则更大的威胁在酝酿，让读者的希望变成更深的不安 | AI 检测"假性好转"模式，在读者以为一切变好时埋入新的危机预兆 |
| **爆发（高潮释放）** | 正义伸张、真相大白、实力碾压——"读者憋着一口气终于吐出来" | AI 确保高潮集中爆发，篇幅不宜过长——"集中爆发比冗长紧张更有效" |
| **留钩（期待重建）** | 爆发过后立即埋入新的钩子，防止读者期待感耗尽后流失 | AI 在每个大事件结尾自动检测是否留下≥1个有效钩子，未达标则提示作者 |

**关键原则**：
- "不要让读者感到憋屈"——如果主角一直被压制，读者情绪会耗尽
- "不要连续给读者出选择题"——每个让读者判断对错的情节，总有一部分人选错。应减少道德选择题，增加价值判断题（"他该不该死？"而非"杀了A对不对？"）
- "宁愿牺牲前期密集爆发，也要保持节奏稳定"——前后节奏落差过大，读者会不适应而弃读

### 角色功能定位规则 🆕

> **番茄方法论启发**：不同角色在文中的定位不同，评估标准也应不同。不能用同一把尺子衡量主角和配角。

| 角色类型 | 核心标准 | 说明 | 系统应用 |
|----------|---------|------|---------|
| **主角** | 功能性 > 完美性 | 主角不必（也不应）完美。性格缺陷（过度自信、躁郁、社恐）恰恰是剧情发动机。主角之所以是主角，是因为他能做到别人做不到的事 | AI 在角色创建时主动建议添加"功能性缺陷"——这个缺陷能引出什么剧情线？ |
| **配角** | 记忆点 > 完整性 | 配角不必全方位介绍，只需一个最独特的记忆标签。"爱吃甜食的暗卫"比"玉树临风的暗卫"更能让读者记住 | AI 为每个配角自动生成≥1个记忆锚点，并检测配角的出场篇幅是否超过了主角 |
| **反派** | 破坏力 > 邪恶值 | 反派可以不够邪恶，但必须能给主角带来实质压力。"修为高超但被心魔控制的大能"——他未必坏，但他随时可能失控造成毁灭性后果 | AI 评估反派压力指数：反派造成的实际/潜在破坏程度，预警"反派太菜"问题 |
| **群像角色** | 独立诉求 > 服务主角 | 每个角色（包括闺蜜、助手）都应有自己的情感诉求和困境，而非只是主角的"工具人" | AI 在角色创建时为每个角色（包括次要角色）生成独立的"欲望+困境"双线 |

## 第三部分：全功能模块清单

### 3.1 核心创作模块

| 模块名称 | 功能描述 |
|---------|---------|
| **大岗孵化器** | 大岗诊断（吸引力/差异化/风险分析）、变体矩阵探索（多维度方向穷举）、对标作品融合分析、关键场景锚点收集、情感基调定调——在进入正式创作前将方向想透 |
| **锚点管理器** | 五类锚点（场景/角色/情节/情感/结构）的创建、编辑、可视化分布图；每步生成前的自动注入与生成后的对齐度检测报告；锚点冲突自动提示 |
| **想法扩写模块** | 多版本扩写、锚点对齐度评分、独特性自评、版本对比 |
| **核心设定模块** | 结构化设定存储（支持角色圣经/世界观规则体系/冲突网络）、设定全局同步（所有生成环节自动引用已确认的设定，避免前后矛盾）、设定导出/导入 |
| **分卷大纲模块** | 分卷可视化拖拽排序、叙事结构节点映射（支持三幕剧/英雄之旅/起承转合等多种结构模板）、分卷进度标记、分卷整体预览 |
| **分章情节模块** | 章节批量生成、因果链自动标注与检测、巧合比例预警、惊喜度评分（1-10）+ 低分自动生成替代版本、钩子自动优化 |
| **章节大纲模块** | 节点化大纲编辑、读者信息差状态标注（读者知道什么 vs 角色知道什么）、情感强度曲线可视化、大纲导出为思维导图 |
| **正文生成模块** | 文风指纹注入、角色声音一致性校验、AI痕迹自动检测与消除、段落级微指令重写、上下文连贯性检测 |
| **全局修改同步** | 双向同步：上游修改可同步下游 / 下游产生新灵感可反向更新上游设定；冲突检测提示 |

### 3.2 叙事记忆体（长篇一致性保障系统）

> 解决 AI 主导创作长篇小说时的"遗忘问题"——确保第80章时 AI 仍记得第3章的伏笔、第12章的角色细节、第25章的世界观规则。

| 子模块 | 功能描述 |
|--------|---------|
| **伏笔-回收追踪器** | 自动扫描所有已生成章节，识别伏笔（暗示、铺垫、未完成事件），追踪回收状态（已回收/待回收/疑似遗忘）。生成新章节时自动将待回收伏笔列表注入 Prompt，确保不遗漏 |
| **角色状态快照引擎** | 每章结束后自动更新每个角色的当前状态：所在位置、心理状态、已知信息、人际关系变化、财产/能力变化。新章节生成前 AI 必须读取所有出场角色的状态快照 |
| **世界观一致性校验** | 核心设定中的世界观规则被编码为校验规则。新章节生成后自动跑校验："是否违反已建立的魔法/科技规则？""是否出现时间线矛盾？""是否符合已设定的地理/政治逻辑？" |
| **时间线引擎** | 维护完整故事时间线，标注每个事件发生的精确时间点（支持相对时间与绝对时间）。自动检测"角色A在同一天出现在两个相隔千里的地点"之类的时间矛盾 |
| **一致性报告** | 每完成一个章节后自动生成一致性摘要：伏笔回收率、角色出场均衡度、时间线完整性、世界观规则违反记录 |

### 3.3 生成-批判双通道

> AI 不仅负责"写"，还必须在每次生成后自动切换为"编辑"视角，对刚生成的内容进行多维度审查。批判结果为后续生成提供校准，也决定内容是否需要提审作者。

| 评估维度 | 检查内容 |
|----------|---------|
| **角色一致性** | 角色行为是否符合档案定义的内在逻辑？对话是否符合该角色的口头禅和句式特征？角色关系动态是否正确？ |
| **情节合理性** | 因果链是否完整？是否存在"机械降神"式解决？人物动机是否充分？ |
| **对话自然度** | 对话是否有分层（表层 vs 潜台词）？多人对话中是否存在角色失声？ |
| **节奏把控** | 场景密度是否合理？是否存在过长的信息倾倒？动作场景与静态场景比例是否恰当？ |
| **锚点对齐** | 所有有效锚点是否在本章内容中正确体现？ |
| **AI 痕迹** | 是否存在 AI 生成文本的典型特征（过度使用"然而/因此/总的来说"、段落结构过于规整、缺乏个性修辞癖好）？ |

**工作流**：生成 → 切换编辑模式自评 → 评分低于阈值的维度自动标记 → 异常项提审作者 / 正常项静默通过

### 3.4 反均值化引擎

> 对抗大语言模型"预测最可能答案"的天然倾向，确保 AI 主导创作的内容保持独特性和意外感。

| 功能 | 说明 |
|------|------|
| **惊喜度量化** | AI 在生成每个情节节点时自评"这个走向的意外程度（1-10）"，低于阈值时自动生成2-3个更具颠覆性的替代版本供作者参考 |
| **类型惯例偏离度** | 系统维护当前作品类型标签，生成时标注"该情节模式在同类小说中出现频率约XX%"，高频模式预警，鼓励作者选择差异化方向 |
| **破坏者模式** | 作者可随时对任一已生成内容开启——AI 提出"最意想不到但合理"的颠覆方案（如"如果这个配角其实是反派卧底""如果这个胜利实际上是陷阱的诱饵"） |
| **多模型辩论** | 同一情节节点用不同模型各生成一个版本，两个版本的差异即为创意可能性空间，作者可从中发现非共识灵感 |

### 3.5 过程干预引擎

| 功能 | 说明 |
|------|------|
| **实时暂停与转向** | AI 生成过程中，作者随时暂停，输入方向调整指令，AI 丢弃当前结果按新方向立即重新生成 |
| **微指令注入** | 对任意生成结果，输入一句话调整指令（"更紧张""加快节奏""增加雨天氛围""用更多短句"），AI 局部重写 |
| **条件式预设规则** | 作者预设全局规则，AI 自动遵守："任何角色死亡情节暂停询问""跳过吻戏""每章情节节点≤5个" |
| **A/B 流生成** | 关键分叉点同时生成两个短版本供选择，选定后继续 |
| **否决即学习** | 否决原因自动记录并进入品味档案，同类问题后续自动规避 |

### 3.6 创作者品味档案

> 作者最核心的资产——品味、判断力、选择模式——在系统中被持续捕获、沉淀、复用。让系统越用越懂作者，跨作品复用创作基因。

| 维度 | 说明 |
|------|------|
| **选择偏好模型** | 记录作者在所有 A/B 选择、版本选择中的倾向，建立偏好模型。后续作品中 AI 自动按偏好权重排序选项 |
| **修改模式学习** | 分析作者编辑操作的类型分布（经常删减什么、改什么、补充什么），新生成时主动应用 |
| **否决原因知识库** | 否决原因标签化（太俗套/不符合角色/节奏太慢/对话不自然/偏离锚点/缺乏惊喜），后续生成自动规避同类输出 |
| **跨作品风格基因** | 从一个作品提取"作者偏好DNA"（包含文风指纹、角色偏好、情节倾向、节奏偏好等多维数据），可导入新作品，让新作品从一开始就带着作者的创作基因 |
| **品味档案导出/导入** | 支持将品味档案导出为文件，跨设备/跨安装迁移作者的创作基因 |

### 3.7 创作辅助工具模块

| 模块名称 | 功能描述 |
|---------|---------|
| **灵感收集板** | 快速记录碎片灵感（一句话画面、角色概念、对话片段、情节念头），支持拖拽分类、标签化、关联到作品，支持碎片合龙（多个相关碎片合并为情节节点）。打开应用即显示，而非藏在子菜单中 |
| **素材库** | 内置人设库、场景库、对话库、情节桥段库，支持用户自定义上传私有素材 |
| **内容检测** | 违禁词检测、重复内容检测、前后设定矛盾检测（如主角年龄一致性自动提示） |
| **写作统计** | 字数统计、创作时长统计、每日产出趋势图、创作高峰时段分析、改稿次数统计、删减字数统计 |
| **版本回溯** | 每一步生成/修改均自动保存版本，支持任意时间点回退、版本对比 |
| **多作品管理** | 支持同时创建多个作品，每个作品独立存储所有创作数据、互不干扰 |
| **导出功能** | 支持导出设定文档、大纲文档、完整正文，导出前自动生成**整体质量报告**（伏笔回收率、角色出场均衡性、情绪曲线合理性、节奏异常点、可能的设定矛盾） |
| **写作障碍应对** | 随机灵感提示、当前情节的5种不同走向建议、角色"如果……会怎样"的思维实验 |

### 3.8 章节质量诊断 🆕

> **番茄方法论启发**：综合多位金番作者的改文方法论、编辑拆解技巧、章节自检流程，形成系统化的章节质量诊断引擎。让 AI 在生成后自动执行"编辑级审核"。

#### 3.8.1 开篇质量检查

| 检查项 | 说明 | 严重程度 |
|--------|------|:---:|
| **首页见山** | 第一章第一页（约200字）是否能快速建立场景和期待？是否出现大段背景介绍/设定堆砌？ | 🔴 致命 |
| **章末留钩** | 每章结尾是否留下≥1个有效钩子？钩子类型：悬念/反转预告/新信息/情感悬念/冲突升级 | 🔴 致命 |
| **金手指有机度** | 金手指是否与剧情有机融合（而非孤立面板）？是否起到了推动剧情而非水字数的作用？ | 🟡 警告 |

#### 3.8.2 章节信息密度诊断

> 核心方法论（煮熟的来福鸽）："读者一句话概括本章=节奏太慢；读者要十几句话才能消化=节奏太快"

| 诊断维度 | 检测方式 | 优化建议 |
|----------|---------|---------|
| **信息过载** | AI 自动摘要本章核心信息点数量，>8个标记为过载 | 拆分为多章或删减次要信息 |
| **信息贫乏** | 核心信息点<2个且字数>2000，标记为"水文风险" | 增加剧情推进或合并章节 |
| **过渡章检测** | 检测章节是否仅有物理场景转换而无新的期待感产生 | 提示："不存在纯过渡章，每章都需制造新的期待感" |
| **结算时刻平衡** | 大高潮后的章节是否合理安排了"结算时刻"（日常变化/配角反应/情感消化），但未过度拖沓 | 结算内容>50%章节字数时预警 |

#### 3.8.3 节奏与情绪曲线诊断

| 检查项 | 说明 |
|--------|------|
| **情绪单调性** | 连续3章情绪强度评分（1-10）无显著变化（波动<2）时预警——读者需要情绪起伏 |
| **压抑持续性** | 连续5章主角处于被动/受压状态时预警——"读者不是来看主角一直被虐的" |
| **高潮疲劳** | 连续3章都处于高强度爆发时预警——"紧凑节奏一旦拉长就有阅读疲惫感" |
| **爆发集中度** | 高潮章节篇幅超过正常章节150%时预警——"集中爆发比冗长紧张更有效" |
| **节奏断层** | 前后两段节奏差异过大（如突然从日更慢热跳到密集爆发）标记为"节奏断层风险" |

#### 3.8.4 冲突强度检测

> 核心方法论（煮熟的来福鸽）："冲突不够的根源是没有绝对的对抗理由"

| 检查项 | 说明 |
|--------|------|
| **对抗动机** | 冲突双方是否有"不能退让的理由"？AI 检测并标注动机强度 |
| **代价具象化** | 主角不行动的代价是否被具象描述？"代价是死"比"会失败"更有力；"死几个"比"死"更具体 |
| **冲突类型分布** | 统计全篇冲突类型（人与人/人与环境/人与自我），单一类型占比>80%时建议多样化 |
| **反派压力指数** | 反派是否给主角带来了实质压力？反派战斗力/智力/资源是否≥主角？ |

#### 3.8.5 改文自检清单

> 核心方法论（周大白 + 煮熟的来福鸽）：系统化的通文改稿步骤

| 步骤 | 操作 | 系统支持 |
|------|------|---------|
| **① 定位问题** | 看数据（章节读完率曲线）或自我判断哪里写偏了 | AI 自动分析章节读完率折线图，标注异常下滑点 |
| **② 分类问题** | 判断是小毒点（一两句话/一两个人物）还是大剧情问题（几章到十几章） | AI 自动分类：微调级/段落级/章节级/结构级 |
| **③ 小毒点处理** | 直接删除，不犹豫 | AI 自动检测评论中的"毒点"关键词（读者愤怒/不适点） |
| **④ 大剧情修改** | 整体推倒重写，紧贴大纲骨架；如果驾驭不了，不强行改，当练笔 | AI 在推倒重写时自动保留核心设定信息 |
| **⑤ 朗读验证** | 嘴巴读出来，有卡顿处修改 | AI 辅助生成朗读友好度评分 |
| **⑥ 独立性检查** | 每章作为独立内容看，是否有看点？是否让人想继续读？ | AI 对每章执行"独立章节评估" |
| **⑦ 换名重读** | 把主角名换成陌生名重读，问自己：如果不是我写的，好看吗？ | AI 模拟"陌生读者视角"评估 |

## 第四部分：文风模块完整设计

> **核心洞察**：AI 主导创作模式下，文风不是"作者选择用什么风格写"，而是"作者希望 AI 用什么风格替自己写"。如果 10 个不同作者用同一系统各生成一本书，这 10 本书读起来必须像 10 个不同的人写的——这是文风模块的最高目标。

### 4.1 核心功能清单

| 功能点 | 详细描述 |
|--------|---------|
| **自定义文风创建** | 用户可新增文风，填写：文风名称、文风描述、文风样例（100-300字）、Prompt模板 |
| **文风库管理** | 支持编辑、删除、克隆、导出/导入操作，内置10+预设文风（免创建直接使用） |
| **作品文风关联** | 为单个作品指定专属文风，该作品下所有生成环节默认应用指定文风，支持随时切换 |
| **文风预览** | 选择文风后可输入一句话测试生成短文本，预览文风效果 |
| **全局/局部文风切换** | 支持全局默认文风、单章节/单生成环节临时指定文风，优先级：**临时指定 > 作品绑定 > 全局默认** |

### 4.2 文风指纹（Style Fingerprint）

> 超越简单的"爽文/细腻"标签分类，从作者的认可样例中提取可量化的多维文风特征，使 AI 生成文本的风格可测量、可约束、可校验。

| 指纹维度 | 说明 | 量化方式 |
|----------|------|---------|
| **平均句长** | 每句话的平均字符数 | 15-25（短句型）/ 26-40（中等）/ 40+（长句型） |
| **句长波动率** | 长短句交替的频率与幅度 | 标准差 + 变异系数 |
| **段落长度偏好** | 典型段落的句子数/字符数 | 短段落型 / 中等 / 长段落型 |
| **对话密度** | 对话占总字数的百分比 | 10-70% |
| **描写密度** | 环境描写、心理描写、动作描写各自占比 | 三维比例分布 |
| **修辞密度** | 比喻/拟人/排比/反问等修辞手法的每千字出现次数 | 低/中/高/极高 |
| **叙述距离** | 与角色的心理距离 | 远距离客观 / 中距离跟随 / 近距离透视 / 内心独白 |
| **词汇偏好** | 高频词汇、偏好句式、禁忌表达 | 词频 Top200 + 自定义黑名单 |
| **节奏特征** | 信息密度、场景切换频率 | 快速/中等/舒缓 |

**文风指纹工作流**：用户提供1-3段认可的样例文本 → 系统自动提取指纹数据 → 生成时注入指纹约束 → 生成后计算偏差度 → 偏差过大时自动标记

### 4.3 AI 痕迹消除

> AI 生成的文本天然带有可识别的"AI 口音"——这会使作品失去个人化的质感。系统需内置检测与消除机制。

| 检测项 | 说明 |
|--------|------|
| **过度连接词** | "然而""因此""总的来说""与此同时""不仅如此"等AI高频连接词的密度检测，超标时自动替换或删减 |
| **段落结构规整化** | AI 倾向于每段"总-分-总"、每段长度接近、缺乏碎片句和单句段落。检测后标记结构过于规整的段落 |
| **情感表达模板化** | "心中涌起一股……""眼中闪过一丝……""嘴角微微上扬"等AI高频情感表达模式的检测与替换建议 |
| **缺乏个人化修辞** | AI 的比喻和意象使用趋向常规化，缺乏个人独特的修辞癖好。系统标注并提示作者注入个人化表达 |

### 4.4 文风稳定性与进化

| 功能 | 说明 |
|------|------|
| **文风稳定性报告** | 全书生成完毕后，对所有章节逐章检测文风指纹偏差，标记文风漂移的章节，生成全书画风一致性热力图 |
| **文风进化支持** | 允许作者在一个作品内有意让文风渐进变化（如前期轻快→中期紧张→后期沉重），系统按预设曲线动态调整文风参数而非全书一刀切 |
| **文风指纹导出** | 将提取的文风指纹数据导出为独立文件，可导入到新作品或分享给其他创作者 |

## 第七部分：多模型统一对接模块设计

### 7.1 核心设计

封装统一调用层，屏蔽不同模型的API差异，上层业务无需关心调用的是哪个模型：

```typescript
// 统一调用接口
interface ModelRequest {
  prompt: string;
  styleId?: number;                                              // 绑定的文风ID，自动注入文风Prompt
  modelType?: 'deepseek' | 'gemini' | 'openai';                 // 不指定则按优先级自动选择
  maxTokens?: number;
  temperature?: number;
}

interface ModelResponse {
  success: boolean;
  content: string;
  error?: string;
}
```

### 7.2 模型适配

| 模型 | 适配要点 |
|------|---------|
| **DeepSeek** | 对接DeepSeek开放平台API，默认使用 `deepseek-chat` 模型，支持长文本生成 |
| **Gemini** | 对接Google Gemini API，默认使用 `gemini-1.5-pro` 模型，支持多模态输入 |
| **OpenAI** | 对接OpenAI API，默认使用 `gpt-4o` 模型，兼容所有兼容OpenAI接口的第三方模型 |

### 7.3 核心能力

- **自动重试机制**：API调用失败自动重试3次，支持降级到优先级更低的模型
- **文风自动注入**：调用时传入文风ID，自动将文风Prompt模板、文风指纹、样例文本拼接到请求中
- **叙事记忆体注入**：生成正文时自动带入角色状态快照、待回收伏笔列表、世界观校验规则、时间线约束
- **上下文自动拼接**：生成正文时自动带上前文内容，保证上下文连贯性
- **费用统计**：自动统计每个模型的调用次数、Token消耗，本地展示使用明细

### 7.4 多模型辩论机制

> 利用不同模型的"认知差异"发掘创意可能性。同一个情节节点用不同模型各生成一个版本，两个版本的差异即是创意的非共识空间。

| 功能 | 说明 |
|------|------|
| **双模型并行生成** | 关键情节节点同时调用两个不同模型（如 DeepSeek + Gemini），各生成一个版本 |
| **差异对比** | 系统自动对比两个版本的差异点：情节走向差异、角色处理差异、文风呈现差异，高亮"非共识"段落 |
| **融合建议** | AI 分析两个版本各自的优势，提出融合建议——"版本A的情感张力更强，版本B的逻辑更严密，建议取A的情感处理+B的因果逻辑" |
| **手动/自动触发** | 作者可对任意节点手动触发辩论模式；系统也可在惊喜度评分<阈值时自动触发，作为反均值化手段之一 |
| **成本控制** | 双模型调用成本翻倍，作者可配置触发条件（仅在惊喜度<4或关键帧节点时启用） |

## 第八部分：火山引擎图片生成功能集成

### 8.1 对接方案

对接火山引擎智能画图API，支持以下能力：

| 功能 | 描述 |
|------|------|
| **多场景生成** | 支持生成作品封面、人物人设图、场景插图、章节配图4种类型 |
| **参数配置** | 用户可自定义图片尺寸（横版/竖版/正方形）、生成数量、风格（写实/二次元/古风等） |
| **本地存储** | 生成的图片自动保存到本地用户目录，关联对应作品/章节，支持导出 |
| **历史记录** | 所有生成的图片自动存档，可随时复用、删除 |

### 8.2 调用流程

1. 用户在设置页配置火山引擎 AccessKey / SecretKey，启用图片生成功能
2. 创作过程中点击"生成图片"，输入Prompt或自动提取当前章节/设定内容作为Prompt
3. 主进程调用火山引擎API提交生成任务，轮询等待生成结果
4. 生成成功后下载图片到本地，存储路径写入 `generated_images` 表，前端展示预览
